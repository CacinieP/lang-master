# Lang Master — 项目愿景与架构

> 从自然语言学与编程哲学的视角，帮助初学者精通编程语言排行榜 Top 10 语言，并掌握超越单一语言的元能力。

## 1. 灵感来源：Typing Master 的设计映射

Typing Master（多语言打字练习桌面应用）的核心设计模式，可以直接迁移到编程语言学习场景：

| Typing Master 概念 | Lang Master 对应概念 | 说明 |
|---|---|---|
| 6 种自然语言 | Top 10 编程语言 | 从 fr/es/it/pt/ru/ja → Python/C/Java/C++/C#/JS/Go/Rust/Swift/R |
| 词汇数据模块 (`words/*.js`) | 语言特性数据模块 (`langs/*.json`) | 每种语言的语法、语义、惯用法数据集 |
| 分类体系 (salutations/nourriture...) | 编程范式分类 (OOP/FP/并发/系统...) | 按概念域组织而非按语言 |
| 难度分级 (A1→C1 / N5→N1) | 编程难度分级 (P1→P5) | 从表达式→模块→并发→元编程 |
| 虚拟键盘多布局 (US Intl/ЙЦУКЕН/JIS) | 编程范式视图切换 (过程式/OOP/函数式) | 同一问题在不同范式下的表达方式 |

## 2. 核心创新：语言学-编程学双层映射

### 2.1 自然语言学五层结构 → 编程语言五层结构

| 语言学层次 | 定义 | 编程语言对应 | 示例 |
|---|---|---|---|
| **语音层 (Phonology)** | 声音的组织规律 | **词法层 (Lexical)** — token/关键字/运算符的合法形式 | `if`, `->`, `===`, `fn` |
| **形态层 (Morphology)** | 词的内部结构与派生 | **语法构造层 (Syntactic Morphology)** — 表达式/语句的内部组合规则 | `a + b`, `x.y()`, `&mut T` |
| **句法层 (Syntax)** | 词如何组合成句子 | **句法层 (Syntax)** — 语句/块/模块的组合规则 | `for..in`, `class X extends Y`, `match x { }` |
| **语义层 (Semantics)** | 句子的意义 | **语义层 (Semantics)** — 程序行为的指称 | 求值规则、类型系统、副作用语义 |
| **语用层 (Pragmatics)** | 语境中的使用 | **语用层 (Pragmatics)** — 设计模式/惯用法/生态系统惯例 | Pythonic, Go's `if err != nil`, Rust's `Result<T,E>` |

**教学法意义：** 初学者通常只学 Syntax（句法），但精通一门语言需要掌握全部五层。Lang Master 将五层结构显式化，让学习者知道自己处于哪一层、还需攻克哪一层。

### 2.2 Sapir-Whorf 假说的编程版

**弱形式（已被广泛接受）：** 编程语言影响（但不决定）程序员的思维模式。

| 语言 → 思维模式影响 | 典型表现 |
|---|---|
| Python → "一切皆对象，简洁优先" | 开发者倾向用最短代码解决问题，关注可读性 |
| C → "手动管理，精确控制" | 开发者对内存布局有直觉，追求零开销抽象 |
| Java → "一切皆类型，安全优先" | 开发者习惯用接口和继承组织复杂关系 |
| Rust → "所有权思维，零妥协安全" | 开发者自然思考资源生命周期，排斥隐式拷贝 |
| Haskell → "纯函数思维，数学优先" | 开发者排斥副作用，优先用组合而非变异 |
| Go → "简洁并发，实用主义" | 开发者用 goroutine 而非复杂抽象处理并发 |

**元能力目标：** 让学习者意识到不同语言的思维框架，并能主动切换——而非被单一语言锁定。

```

Algol 语族 (≈ 印欧语系)
├── C 分支 (≈ 感谢语族日耳曼语支)

├── ML/Haskell 分支 (≈ 印欧语系印度-伊朗语支)
│   └── (间接影响 Rust/Swift/Scala)

│   └── (间接影响 Ruby/Objective-C/Swift)
```

### 5.2 核心交互：概念级打字 → 概念级编程

Typing Master 是「看到单词 → 打出字符」。
Lang Master 是「看到概念 → 用指定语言表达」。

示例：
```
概念提示：  "对一个集合进行过滤并映射"
─────────────────────────────────────
Python:    [f(x) for x in items if p(x)]
JavaScript: items.filter(p).map(f)
Rust:      items.iter().filter(p).map(f)
Go:        // 没有内置filter/map，需要手写循环
C:         // 纯手动for循环+函数指针
Java:      items.stream().filter(p).map(f).collect()
```

**关键洞察：** 同一概念在不同语言中有不同的"表达密度"——这就是编程语言的 Sapir-Whorf 效应。

Typing Master 的虚拟键盘显示"下一个要按的键"。

## 7. 七项设计约束

### C1: 范式优先，语言次之

**规则：** 所有内容按概念/范式组织，绝不按语言组织。不存在"Python 第1章、Python 第2章"这种结构。

**Why：** 按语言组织会强化 Sapir-Whorf 锁定——学习者把 Python 当作"我的语言"，其他语言当作"外语"。按概念组织则让学习者始终在对比中理解，如同学习法语和西班牙语时同时看两个语言的同一个语法点，而非先精通法语再看西班牙语。

**How to apply：** 数据结构设计时，每个概念节点包含10种语言的表达方式，而非每种语言有自己的概念列表。UI导航以"变量→函数→类型→并发→元编程"为路径，不以"Python→Java→Rust"为路径。

### C2: c+1 原则（Krashen i+1 适配版）

**规则：** 每次练习仅引入恰好一个超出当前水平的新概念。绝不一次抛出多个新概念。

**Why：** Krashen 输入假说证明自然语言习得最有效的输入是 i+1（略高于当前水平），而非 i+5。编程学习的认知负荷更重——同时引入所有权、生命周期、trait、泛型会让学习者崩溃。Rust 学习曲线陡峭正是因为传统教程违反了此原则。

**How to apply：** 难度分级不是"语言难度"而是"概念密度"——P1 级练习每题最多1个新概念，P2 级每题最多2个已知概念+1个新概念。挑战模式可以增加时间压力但不增加概念密度。

### C3: 负迁移必须显式标注

**规则：** 当已知语言 A 的习惯会在语言 B 中产生错误时，必须在学习者接触 B 之前标注此陷阱。

**Why：** 二语习得研究证明负迁移是最大学习障碍——不是因为目标语言难，而是因为源语言的"正确习惯"在目标语言中变成"错误行为"。Java 的 null 检查习惯 → Rust 没有 null（困惑），Python 的缩进思维 → Go 的花括号（不适），JS 的隐式类型转换 → C 的严格类型（崩溃）。不标注就是让学习者用试错法发现陷阱——极其低效。

**How to apply：** 每个概念节点附带 `traps` 字段，列出从其他语言的负迁移陷阱。UI 在学习者切换语言时弹出"陷阱提示卡"。例如：从 Java 切到 Rust 时提示 "Java: null → Rust: Option<T>，不要用 null 检查，用 match Ok/Err"。

### C4: 三语言最低呈现

**规则：** 任何概念至少用3种语言呈现。绝不只展示一种语言的孤立解法。

**Why：** 单语言呈现会制造"这是唯一做法"的错觉——Sapir-Whorf 锁定的根源。三种语言的最低对比强制学习者发现：(1) 同一概念有多种表达方式；(2) 不同语言的"表达密度"不同；(3) 某些语言天然更适合某些概念。这直接训练范式切换元能力。

**How to apply：** 概念数据中每个节点至少3个语言的表达示例。UI 中的范式视图同时显示3列（如：过程式列/OOP列/FP列）。学习者可以切换查看更多语言，但核心展示始终是3。

### C5: 五层始终可见

**规则：** 界面始终显示当前概念处于语言学五层结构的哪一层（词法→构造→句法→语义→语用）。学习者永远知道自己操作在哪个深度。

**Why：** 初学者最大的困惑不是"怎么写代码"，而是不知道自己在学什么——是在学关键字拼写（词法层）？还是在学表达式组合规则（构造层）？还是在学程序实际行为（语义层）？大多数教程混合五层但不标注层级，学习者无法建立结构化认知。标注五层让学习从"混沌积累"变为"分层攻克"。

**How to apply：** UI 左侧固定显示五层进度条（类比 Typing Master 的实时统计栏）。当前概念高亮所在层级。完成某层所有概念后该层级标记为"已掌握"。统计面板按五层分别展示掌握率而非笼统的"已学 X 个概念"。

### C6: 禁止纯模式匹配通过

**规则：** 学习者不能仅靠语法模式匹配通过练习。每个概念级校验必须包含语义理解验证环节。

**Why：** Typing Master 的核心是打字速度——你可以在不理解词义的情况下高速打字。但 Lang Master 的目标是概念理解而非代码生成速度。如果学习者只是记住了 `items.filter(p).map(f)` 的模式而不理解"filter 是过滤、map 是变换、组合是先过滤再变换"，那他只是获得了一个可复制片段而非元能力。这是编程教育最普遍的失败模式。

**How to apply：** 每个练习分两步：(1) 代码表达（写出/选择正确代码）；(2) 语义验证（解释这段代码实际行为、选择正确输出、识别等价表达式）。只有两步都通过才算完成。精通模式要求三步全对（代码+语义+等价辨识）。

### C7: 无特权语言

**规则：** 10种语言均为平等公民。不存在"参考语言"或"默认语言"。Python 不比 C 更"正确"，Java 不比 Rust 更"标准"。

**Why：** 多数编程教程以一种语言为"主语言"，其他语言为"对比语言"——这隐含了"主语言的表达是标准，其他语言是变体"的假设。从语言学角度看这是荒谬的——法语不是西班牙语的"变体"，它们是平行的语言系统。同样，Python 的列表推导不是 Java Stream 的"简化版"——它们是各自范式下的同等表达。特权语言设置会强化 Sapir-Whorf 锁定。

**How to apply：** 概念展示时随机排列语言顺序（而非总是 Python 先）。不使用"Python写X，其他语言也要写X"的表述，而是"10种语言各有表达X的方式"。数据结构中所有语言字段对称设计，无 `primary` 标记。

### 约束交叉验证

| 约束 | 与其他约束的关系 |
|---|---|
| C1(范式优先) | 必须配合C4(三语言最低)和C7(无特权语言)——否则范式对比会退化为"Python的范式 vs 其他" |
| C2(c+1) | 必须配合C5(五层可见)——学习者才知道"这1个新概念属于哪一层" |
| C3(负迁移) | 依赖C7(无特权语言)——如果Java是特权语言，Java→Rust的负迁移不会被标注 |
| C4(三语言) | 必须配合C1(范式优先)——否则三种语言可能都是同范式，失去对比意义 |
| C5(五层) | 必须配合C6(禁止模式匹配)——语义层验证是反模式匹配的关键 |
| C6(禁止匹配) | 依赖C5(五层)和C2(c+1)——语义验证不能超出当前概念层级 |
| C7(无特权) | 是C1+C3+C4的根基——特权语言会破坏范式对比、负迁移标注、三语言呈现的对称性 |
